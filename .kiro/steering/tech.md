# 技術スタック — place-finder

## フロントエンド

### React を採用する理由

- エコシステムが成熟しており、地図系ライブラリ（Google Maps React wrapper 等）との統合実績が豊富
- コンポーネント指向の設計が FSD アーキテクチャと相性が良い
- チームの学習コストが低く、MVP を素早く立ち上げられる

### Next.js ではなく React（Vite）を選ぶ理由

- place-finder は位置情報ベースの SPA であり、SSR/SSG の恩恵が薄い
- SEO 要件が低い（ユーザーの現在地に依存する動的コンテンツが中心）
- Vite の高速な開発体験と、シンプルな構成を優先する
- バックエンドは Elysia で独立しており、フルスタックフレームワークの統合は不要

### React Query を使う理由

- サーバー状態の管理（キャッシュ、再取得、ローディング状態）を宣言的に扱える
- API クライアント生成物との組み合わせで、型安全なデータ取得レイヤーを構築できる
- 手動での状態管理コードを大幅に削減し、MVP の開発速度を上げる

### Valibot を使う理由（フロントエンド専用）

- フロントエンド側のスキーマ/バリデーションライブラリとして採用する
- バンドルサイズが極めて小さく、SPA のパフォーマンスに影響しない
- TypeScript ファーストで型推論が強力
- フォーム入力の検証、API レスポンスのランタイムバリデーションに使用する
- Zod と同等の表現力を持ちつつ、ツリーシェイキングに優れる
- バックエンドのスキーマライブラリ（TypeBox）とは役割が異なる。フロントエンドは Valibot、バックエンドは TypeBox で統一する

### `packages/api-client` の生成物を利用する前提

- Orval を用いて、バックエンドの OpenAPI スキーマから API クライアントと型を自動生成する
- 生成物は `@place-finder/api-client` パッケージとして `packages/api-client/generated/` に出力する
- Orval は React Query 用のカスタムフック生成をサポートしており、生成物をそのまま React Query のクエリ関数として利用できる
- フロントエンドは生成物を import して利用する。手書きの API 呼び出しコードや型定義は作成しない

### FSD（Feature-Sliced Design）を採用する理由

- 機能単位でコードを整理でき、place-finder のような機能追加型プロダクトと相性が良い
- レイヤー構造により依存方向が明確になり、コードの見通しが良くなる
- 初期は薄く使い、機能が増えた段階で段階的に厳密化できる柔軟性がある

### Mantine UI を採用する理由

- 豊富なコンポーネント群を持ち、MVP で必要な UI（リスト、カード、フィルタ、モーダル等）を素早く構築できる
- Headless ではなくスタイル付きコンポーネントを提供するため、デザイン工数を大幅に削減できる
- CSS Modules ベースでスタイルが閉じており、グローバル汚染が起きにくい
- フォーム連携（`@mantine/form`）が充実しており、Valibot と組み合わせたバリデーションが容易
- アクセシビリティが標準で考慮されている
- React 19 対応が進んでおり、エコシステムとの互換性が高い
- 地図表示以外の UI 要素（検索バー、ジャンルフィルタ、店舗カード、レイアウト等）を効率的に実装できる

## バックエンド

### Elysia を採用する理由

- Bun ネイティブで動作し、ランタイムを統一できる
- TypeScript ファーストで、型安全なルーティングとバリデーションを標準で提供
- OpenAPI スキーマの自動生成（`@elysiajs/swagger`）が組み込みでサポートされている
- 軽量で起動が速く、MVP の開発サイクルに適している

### FastAPI ではなく Elysia を選ぶ理由

- フロントエンド（TypeScript）とバックエンドの言語を統一し、モノレポ内での型共有・ツール共有を容易にする
- Bun をランタイムとして統一することで、開発環境の複雑さを抑える
- FastAPI は Python エコシステムの強みがあるが、このプロダクトの初期段階では ML/AI 処理が不要であり、言語統一のメリットが上回る
- 将来 AI 機能を追加する場合も、Python サービスを別プロセスとして連携する選択肢がある

### TypeBox を採用する理由（バックエンド専用）

- バックエンド側のスキーマ/バリデーションライブラリとして採用する
- Elysia が TypeBox をネイティブサポートしており、ルートのリクエスト/レスポンス型定義に直接利用できる
- TypeBox で定義したスキーマが JSON Schema に変換され、そのまま OpenAPI ドキュメントに反映される
- つまり TypeBox のスキーマ定義が OpenAPI の Single Source of Truth となる
- バックエンドファーストで堅牢な API 契約を定義するための基盤技術

### OpenAPI を前提にする理由

- Elysia + TypeBox の組み合わせにより、コードから正確な OpenAPI スキーマが自動生成される
- スキーマからフロントエンド用の API クライアントと型を自動生成し、手動同期を排除する
- API ドキュメントが常に実装と同期した状態を維持できる
- バックエンドファーストの設計フロー（後述）により、API 契約の品質を担保する

### バックエンドファースト OpenAPI 設計方針

- API 設計は必ずバックエンドの実装を起点とする。フロントエンドが先に型を定義することはしない
- 設計フロー: TypeBox スキーマ定義 → Elysia ルート実装 → OpenAPI スキーマ自動生成 → Orval でクライアント生成 → フロントエンドで利用
- バックエンドの TypeBox スキーマが API 契約の唯一の信頼源（Single Source of Truth）
- API の変更は必ずバックエンドから行い、生成パイプラインを通じてフロントエンドに伝播させる
- フロントエンドが独自に API の型を手書きすることは禁止する

### モジュラーモノリス構成を採用する理由

- MVP 段階ではマイクロサービスは過剰であり、デプロイ・運用の複雑さを避ける
- 機能ごとにモジュール境界を設けることで、将来的な分離が容易になる
- 単一プロセスで動作するため、開発・デバッグが簡単

### 各モジュールを `src` 直下に置く理由

- ネストを浅く保ち、ファイル探索のコストを下げる
- `src/places/`, `src/recommendation/` のように機能名がそのままディレクトリ名になり、直感的
- 過剰なレイヤー分割（`src/modules/places/...`）を避け、MVP に適したフラットな構成を維持する

### 初期段階で DB や認証を過剰に広げない理由

- MVP はユーザーアカウント不要で、外部 API（Places API）のプロキシ＋推薦ロジックが中心
- DB は推薦パラメータの管理やキャッシュ等、必要になった時点で導入する
- 認証は将来のアカウント機能追加時に導入する
- 最初から入れると、開発・テスト・デプロイの複雑さが不必要に増す

## モノレポ

### モノレポを採用する理由

- フロントエンド・バックエンド・API クライアントが密結合するプロダクトであり、同一リポジトリで管理する方が変更の整合性を保ちやすい
- Bun workspaces によるシンプルなパッケージ管理で十分であり、Turborepo 等の追加ツールは現時点では不要
- CI/CD、リンター、フォーマッター（Biome）の設定を一元管理できる

### 各パッケージの責務

| パッケージ | 責務 |
|---|---|
| `apps/web` | ユーザー向け SPA。UI 表示、ユーザー操作、データ取得を担当 |
| `apps/api` | バックエンド API サーバー。外部 API 連携、推薦ロジック、データ加工を担当 |
| `packages/api-client` | OpenAPI スキーマから自動生成された API クライアントと型定義を提供 |

### なぜこの 3 つから始めるのか

- Web アプリ + API サーバー + 型共有という最小構成で、MVP に必要な全レイヤーをカバーできる
- これ以上パッケージを増やすと、初期段階では管理コストが見合わない

### 共通化を最初からやりすぎない理由

- 「共通化すべきもの」は実際にコードが重複してから判断する方が精度が高い
- 早すぎる抽象化は、不適切な依存関係を生み、後の変更を困難にする
- MVP 段階では `packages/api-client` 以外の共通パッケージは不要と判断する

## API クライアント生成

### Orval による API クライアント自動生成の意義

- バックエンドの TypeBox スキーマ → OpenAPI → Orval という一気通貫のパイプラインで、フロントエンドの型を自動同期する
- 手書きの型定義や API 呼び出しコードの保守コストを完全に排除する
- API の変更がフロントエンドのコンパイルエラーとして即座に検出される

### Orval を選ぶ理由

- React Query 用のカスタムフック（useQuery / useMutation）を直接生成できる
- OpenAPI 3.x に対応し、Elysia が生成するスキーマとの互換性が高い
- 生成コードのカスタマイズ性が高く、axios / fetch / custom instance を選択できる
- モック生成機能があり、バックエンド未実装の段階でもフロントエンド開発を進められる

### フロントエンドとバックエンドの型同期におけるメリット

- TypeBox スキーマ定義 → Elysia ルート実装 → OpenAPI スキーマ自動生成 → Orval でクライアント＆型生成 → フロントエンドで利用
- 「API を変更したら型が壊れる」という安全ネットが自動的に機能する
- バックエンドファーストにより、API 契約の品質がパイプライン全体の品質を決定する

### 生成パイプラインの運用方針

- `packages/api-client/` に Orval の設定ファイル（`orval.config.ts`）を配置する
- `bun run generate` で OpenAPI スキーマの取得 → クライアントコード生成を一括実行する
- 生成物は `packages/api-client/generated/` に出力し、Git 管理する
- CI でスキーマと生成物の整合性をチェックする
- 生成物は手動編集しない。カスタマイズが必要な場合は、生成物をラップするレイヤーを別途設ける
- 開発フロー: バックエンドで API を実装・変更 → `bun run generate` → フロントエンドで生成物を利用
