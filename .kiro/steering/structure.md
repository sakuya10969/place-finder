# アーキテクチャ方針 — place-finder

## フロントエンド（apps/web）

### FSD の採用粒度

FSD（Feature-Sliced Design）は段階的に導入する。初期段階では全レイヤーを厳密に分けず、以下の実用的なサブセットから始める。

```
src/
├── app/          # アプリ全体の初期化、プロバイダー、ルーティング
├── pages/        # ページ単位のコンポーネント（ルートに対応）
├── features/     # ユーザー操作を伴う機能単位（検索、フィルタ等）
├── entities/     # ドメインモデルに対応する表示・操作（店舗カード等）
├── shared/       # 共通ユーティリティ、UI 部品、API クライアント設定
```

### 初期段階での過剰分割を避ける方針

- `widgets/` レイヤーは初期段階では導入しない。`features/` と `pages/` で十分カバーできる
- `processes/` レイヤーも不要。複雑なフロー制御が必要になった時点で検討する
- 各レイヤー内のスライス（機能単位のディレクトリ）も、最初は 2〜3 個で十分
- 「迷ったら `shared/` に置く」を初期の判断基準とし、重複が見えてからリファクタする

### API クライアント生成物の利用方法

- Orval が `packages/api-client/generated/` に React Query 用カスタムフックと型定義を生成する
- `@place-finder/api-client` を `shared/api/` 経由で利用する
- Orval が生成した React Query フックをそのまま `features/` または `entities/` 内で利用する
- 生成物を直接コンポーネントから呼ばず、必ずフック経由でアクセスする
- 手書きの API 呼び出しコードや型定義は作成しない。すべて Orval の生成物を利用する

### Mantine UI の利用方針

- `@mantine/core` をベースに、レイアウト・フォーム・表示系コンポーネントを構築する
- Mantine コンポーネントは `shared/ui/` にラップせず、各レイヤー（`pages/`, `features/`, `entities/`）で直接利用する
- 過剰なラッパーコンポーネントは作らない。Mantine のコンポーネントをそのまま使い、プロジェクト固有の UI パターンが明確になった段階で共通化を検討する
- テーマ設定は `app/` レイヤーの MantineProvider で一元管理する
- `@mantine/form` を Valibot と組み合わせてフォームバリデーションを実装する
- 地図表示は Mantine の管轄外。Google Maps 用のコンポーネントは `features/` 内で独立して管理する

### 責務分離の考え方

| 責務 | 配置 | 方針 |
|---|---|---|
| UI 表示 | `pages/`, `features/`, `entities/`, `shared/ui/` | コンポーネントは表示に専念。ロジックはフックに分離 |
| データ取得 | `features/*/api/`, `entities/*/api/` | React Query フックとして実装。API クライアント生成物を利用 |
| 状態管理 | React Query + ローカル state | グローバル状態管理ライブラリは初期段階では導入しない。React Query のキャッシュをサーバー状態の管理に使い、UI 状態は useState/useReducer で十分 |
| バリデーション | Valibot スキーマ | API レスポンスのランタイム検証、フォーム入力の検証に使用 |

## バックエンド（apps/api）

### モジュラーモノリスの捉え方

- 「将来分離可能なモジュール境界を持つ、単一デプロイのアプリケーション」として捉える
- マイクロサービスの準備ではなく、コードの整理手法として採用する
- モジュール間の依存は最小限にし、直接 import よりも明示的なインターフェースを意識する

### `src` 直下の構成方針

```
src/
├── index.ts          # エントリポイント。Elysia アプリの初期化とモジュール登録
├── places/           # 店舗検索モジュール（Places API 連携）
│   ├── route.ts      # ルーティング定義
│   ├── schema.ts     # TypeBox スキーマ定義（リクエスト/レスポンス）
│   ├── service.ts    # アプリケーションロジック
│   └── client.ts     # 外部 API クライアント（Places API）
├── recommendation/   # 推薦ロジックモジュール
│   ├── route.ts
│   ├── schema.ts
│   ├── service.ts
│   └── scorer.ts     # 重みづけスコアリングロジック
└── shared/           # モジュール横断の共通処理（エラーハンドリング、共通スキーマ等）
```

### 各モジュールの責務境界

- 各モジュールは「1 つのドメイン領域」に対応する
- モジュールは自身のルーティング、ロジック、外部連携を内包する
- モジュール間の依存が必要な場合は、`service` 層の関数を明示的に import する（初期段階では直接 import で十分。インターフェース抽象化は過剰）

### 責務の分け方

| 責務 | ファイル | 方針 |
|---|---|---|
| ルーティング | `route.ts` | Elysia のルート定義。TypeBox でリクエスト/レスポンスのスキーマを定義し、バリデーションと OpenAPI スキーマ生成を同時に実現する |
| スキーマ定義 | `schema.ts` | TypeBox によるリクエスト/レスポンスの型定義。このスキーマが OpenAPI ドキュメントの Single Source of Truth となる |
| アプリケーションロジック | `service.ts` | ビジネスルール、データ加工、モジュール間の調整 |
| 外部 API アクセス | `client.ts` | 外部 API の呼び出しとレスポンスの変換。外部 API の詳細をモジュール内に閉じ込める |

### TypeBox によるバックエンドファースト設計

- 各モジュールの `schema.ts` に TypeBox でリクエスト/レスポンスのスキーマを定義する
- `route.ts` で Elysia の `body()`, `query()`, `response()` に TypeBox スキーマを渡すことで、バリデーションと OpenAPI 定義を同時に実現する
- TypeBox スキーマが API 契約の唯一の信頼源であり、ここを変更すると OpenAPI → Orval → フロントエンドの型に自動伝播する
- バックエンドの API 実装が完了してから `bun run generate` でフロントエンドのクライアントを生成する。この順序を必ず守る

### クリーンアーキテクチャに寄せすぎない理由

- MVP 段階ではレイヤーが少なく、Repository パターンや DI コンテナは過剰
- `route → service → client` の 3 層で十分な関心の分離が得られる
- 抽象化は「同じインターフェースで差し替えたい」具体的な理由が出てから導入する

### 将来の機能追加に備える最低限のルール

1. 新機能は `src/` 直下に新しいモジュールディレクトリを作る
2. モジュールのルートは `route.ts` で定義し、`index.ts` で Elysia アプリに `.use()` で登録する
3. 外部 API の呼び出しは必ず `client.ts` に閉じ込め、service から直接 HTTP リクエストを投げない
4. モジュール固有の型は各モジュール内に定義する。共通型のみ `shared/` に置く

## システム全体

### `apps/web` と `apps/api` の関係

- `apps/web` は SPA として動作し、`apps/api` に対して HTTP リクエストを送る
- 開発時は Vite の proxy 設定（`/api` → `localhost:3000`）で接続する
- 本番環境ではリバースプロキシまたは同一ドメインでの配信を想定する
- フロントエンドはバックエンドの内部実装に依存せず、OpenAPI で定義された契約のみに依存する

### `packages/api-client` の位置づけ

- バックエンドの OpenAPI スキーマから Orval で自動生成された「契約の実体化」
- フロントエンドとバックエンドの間に位置する型安全な橋渡し役
- 手動編集は禁止。変更はバックエンドの TypeBox スキーマ定義を起点とする
- Orval の設定ファイル（`orval.config.ts`）をこのパッケージ内に配置する

### バックエンドファースト生成パイプライン

API の追加・変更時は、以下の順序を必ず守る。

1. バックエンドの `schema.ts` に TypeBox でリクエスト/レスポンススキーマを定義する
2. `route.ts` でスキーマを Elysia ルートに適用し、API を実装する
3. Elysia が OpenAPI スキーマを自動生成する（`@elysiajs/swagger`）
4. `bun run generate` で Orval を実行し、`packages/api-client/generated/` にクライアントコードと型を生成する
5. フロントエンドで生成物を import して利用する

この順序を逆転させない。フロントエンドが先に型を定義したり、手書きの API クライアントを作成することは禁止する。

### Google Maps Platform / Places API の責務配置

- 外部 API の呼び出しはバックエンドの `places/client.ts` に閉じ込める
- API キーの管理はバックエンド側で行い、フロントエンドには露出させない
- フロントエンドの地図表示（Maps JavaScript API）は `apps/web` 側で直接利用する（表示専用のため）
- 店舗データの取得・加工はすべてバックエンド経由とする

### 推薦ロジックの配置

- 推薦ロジックは `apps/api/src/recommendation/` モジュールに配置する
- `scorer.ts` にスコアリングアルゴリズム（距離・評価・営業状況等の重みづけ）を実装する
- スコアリングのパラメータ（重み係数等）は設定値として外出しし、コード変更なしで調整可能にする
- `places` モジュールから取得した店舗データを受け取り、スコア付きで返す純粋な関数として設計する

### 将来の AI 拡張に備える設計

- 推薦ロジックを `recommendation/service.ts` のインターフェースとして切り出しておく
- 現在は `scorer.ts`（ルールベース）が実装を担当する
- AI 推薦を導入する際は、同じインターフェースを満たす新しい実装（例: `ai-scorer.ts`）を追加し、service 層で切り替える
- この差し替えを容易にするため、推薦ロジックは `places` モジュールに埋め込まず、独立したモジュールとして維持する
- AI 処理が Python 等の別言語で実装される場合は、別プロセスとして起動し、`recommendation/client.ts` 経由で HTTP 通信する構成を想定する
