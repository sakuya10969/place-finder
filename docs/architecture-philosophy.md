# アーキテクチャ思想 — place-finder

## 全体方針

- 過剰設計を避け、MVP を素早く作れる現実的な設計を優先する
- 「今やるべきこと」と「今はやらないこと」を明確に分ける
- 将来の拡張余地は残しつつ、早すぎる抽象化はしない

## フロントエンド — FSD（Feature-Sliced Design）

### 基本思想

機能単位でコードを整理し、依存方向を明確にする。ただし初期段階では全レイヤーを厳密に分けず、実用的なサブセットから始める。

### レイヤー構成

```
src/
├── app/          # アプリ全体の初期化、プロバイダー、ルーティング
├── pages/        # ページ単位のコンポーネント（ルートに対応）
├── features/     # ユーザー操作を伴う機能単位（検索、フィルタ等）
├── entities/     # ドメインモデルに対応する表示・操作（店舗カード等）
├── shared/       # 共通ユーティリティ、UI 部品、API クライアント設定
```

### 初期段階の判断基準

- `widgets/`, `processes/` レイヤーは導入しない。必要になった時点で検討する
- 各レイヤー内のスライスは最初は 2〜3 個で十分
- 迷ったら `shared/` に置き、重複が見えてからリファクタする

### 責務分離

- コンポーネントは表示に専念し、ロジックはフックに分離する
- データ取得は React Query フック経由。Orval 生成物を利用する
- グローバル状態管理ライブラリは初期段階では導入しない。React Query のキャッシュ + useState/useReducer で十分
- バリデーションは Valibot スキーマで行う

### Mantine UI の利用方針

- 各レイヤーで Mantine コンポーネントを直接利用する。過剰なラッパーは作らない
- テーマ設定は `app/` レイヤーの MantineProvider で一元管理する
- プロジェクト固有の UI パターンが明確になった段階で共通化を検討する

## バックエンド — モジュラーモノリス

### 基本思想

「将来分離可能なモジュール境界を持つ、単一デプロイのアプリケーション」として構成する。マイクロサービスの準備ではなく、コードの整理手法として採用する。

### モジュール構成

```
src/
├── index.ts          # エントリポイント
├── places/           # 店舗検索モジュール
│   ├── route.ts      # ルーティング定義
│   ├── schema.ts     # TypeBox スキーマ定義
│   ├── service.ts    # アプリケーションロジック
│   └── client.ts     # 外部 API クライアント
├── recommendation/   # 推薦ロジックモジュール
│   ├── route.ts
│   ├── schema.ts
│   ├── service.ts
│   └── scorer.ts     # 重みづけスコアリング
└── shared/           # モジュール横断の共通処理
```

### 責務の分け方

- `route.ts`: Elysia ルート定義 + TypeBox スキーマによるバリデーション
- `schema.ts`: TypeBox によるリクエスト/レスポンス型定義（OpenAPI の Single Source of Truth）
- `service.ts`: ビジネスルール、データ加工
- `client.ts`: 外部 API 呼び出し（外部 API の詳細をモジュール内に閉じ込める）

### クリーンアーキテクチャに寄せすぎない理由

- MVP 段階では `route → service → client` の 3 層で十分
- Repository パターンや DI コンテナは過剰
- 抽象化は「差し替えたい」具体的な理由が出てから導入する

### 拡張ルール

1. 新機能は `src/` 直下に新しいモジュールディレクトリを作る
2. ルートは `route.ts` で定義し、`index.ts` で `.use()` 登録する
3. 外部 API 呼び出しは `client.ts` に閉じ込める
4. モジュール固有の型はモジュール内に定義。共通型のみ `shared/`

## バックエンドファースト設計

- API 設計は必ずバックエンドの TypeBox スキーマ定義を起点とする
- TypeBox → Elysia ルート → OpenAPI 自動生成 → Orval でクライアント生成 → フロントエンドで利用
- フロントエンドが先に型を定義したり、手書きの API クライアントを作成することは禁止
- この一方向のフローにより、API 契約の品質と整合性を担保する

## 推薦ロジックの設計思想

- 推薦ロジックは `recommendation` モジュールとして独立させる
- `places` モジュールから取得した店舗データを受け取り、スコア付きで返す純粋な関数として設計する
- 将来 AI 推薦を導入する際は、同じインターフェースを満たす新しい実装に差し替える
- この差し替え容易性のために、推薦ロジックを `places` に埋め込まず独立モジュールとして維持する
